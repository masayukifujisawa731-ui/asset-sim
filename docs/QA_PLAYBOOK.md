# QA / レビュー プレイブック

ローカルWebアプリ（特に計算・シミュレーション系、React/TS）の品質レビューを
**AIに高速実行させる**ための観点・チェック・対策集。本プロジェクト（資産形成
シミュレーター）で実際に行ったレビューを一般化したもの。

---

## AIへの指示（このファイルの使い方）

1. 下の「観点」を**上から順に**スキャンする。各観点の grep 例・チェック項目を実行。
2. 見つけた問題は **深刻度（🔴実害/クラッシュ・🟠正確性・🟡誤解誘発・⚪仕上げ）** を付けて列挙。
3. 直せるものは**最小差分**で修正し、`tsc → test → lint → build` で必ず検証。
4. **「人手が必要」と明記された項目**（専門家判断・実機/AT検証）は勝手に断定せず、
   指摘として報告する。
5. ドメイン値の既定変更・ライセンス等、**ユーザー判断が要るもの**は確認してから。
6. 各修正は意味単位でコミット。プロパティ/UIテストで回帰を固定する。

## 観点1: 数値の頑健性（計算系では最優先）

純関数テストは「実装が出した値の再現」になりがちで**正しさを証明しない**。
不変条件で殴るのが効く。

**探し方 / チェック**
- 境界値：開始>終了、0/負の入力、極大値、空配列、重複、超長期。
- 縮退：`/` や `Math.log`/`Math.pow` の分母0・真数0・`1+x<=0`。
  ```
  grep -rn "Math.pow\|Math.log\|/ i\|/ (" src/lib
  ```
- `rows[rows.length-1]` 等、**空配列前提のアクセス**（→クラッシュ源）。
- 永続化値（localStorage/URL）を**型検証せず**計算に流していないか。

**対策（本PJ実例）**
- ループが0回→空配列→`rows[last]`参照で**画面真っ白**だった → 最低1行を保証。
- `fvCoeff` が極小利回りで `(1+r)^(1/12)-1` が0に丸まり **Infinity** → `log1p/expm1`
  化＋`i===0`で係数12に縮退。**fast-checkが発見**。
- `sanitize()` 層を新設し、保存値を型強制・範囲クランプ・相互制約（end≥start）を保証。
- **プロパティテスト**（fast-check）：全出力が有限／恒等式（合計=内訳和）／単調性
  （入力順→出力順）／分位の順序 p10≤…≤p90。

**人手**: 不要（自動化可能）。

## 観点2: ドメイン/モデルの正確さ・誠実さ

計算ツールの最大の失敗は「もっともらしい嘘の数字」。

**チェック**
- 税・制度・上限などの**定数が現行法と合っているか**、年度切替はあるか。
- 名目 vs 実質（インフレ）、税引き前 vs 後を**混同していないか**。
- 「純増」「確実」等、**メリットを誇張する表現**。未計上項目の注記。
- 上限・枠（年間/生涯など）を**取りこぼしていないか**。

**対策（本PJ実例）**
- インフレ実質表示トグルを追加（名目値の過大印象を是正）。
- NISA**年間枠**の未考慮を修正（生涯枠だけ見ていた）。
- iDeCo「純増」表記を弱め、**出口課税で目減りする旨**を明記。

**人手**: 数値・前提の妥当性は**専門家レビュー推奨**（税・FP等）。AIは簡略化箇所を列挙するに留める。

## 観点3: 固有状態（ペルソナ依存）のハードコード検出

サンプル/初期ペルソナの値が**既定や固定値として残留**しがち。

**探し方**
```
grep -rn -E "[0-9]{2,}歳|[0-9],?[0-9]{3}万|= ?[0-9]{4,}" src --include=*.ts*
```
- 編集可能なはずの目標額・年齢・金額が**文言にベタ書き**されていないか。
- 判定基準（例：特定年齢）が**入力レンジ外でも固定**になっていないか。
- 未使用のデッド定数。

**対策（本PJ実例）**
- 「NISA既消化60万」ハードコード→**編集可能フィールド（既定0）**化。
- 説明文の「3,000万・5,000万」ベタ書き→ `targetA/targetB` 動的化。
- MC確率の判定年齢 45/50/55 固定→**年齢レンジに連動**。
- 未使用の `EXIT_TAX_RATE` 削除。

**人手**: 既定値の妥当性はユーザー確認。

## 観点4: プライバシー / セキュリティ

**探し方 / チェック**
- 外部通信・CDN・Webフォント・解析の有無：
  ```
  grep -rn -E "fetch\(|XMLHttpRequest|https?://|cdn|googleapis|analytics" src index.html
  ```
- 機微データ（個人情報・金額）の **localStorage 平文保存**、消去手段の有無。
- `npm audit` の結果と**実害有無の判断**（使い方次第で非該当なことも）。
- 静的配信時の **CSP**。

**対策（本PJ実例）**
- 完全ローカルを確認の上、**CSPをビルド時のみ注入**（devはHMRが
  inline/eval/wsを使うため適用しない＝`apply:'build'`）。
- 「保存データを消去」ボタン＋「このブラウザにのみ保存・外部送信なし」明示。
- `index.html`：`lang`、タイトル、`description`、`robots noindex`。
- 監査勧告は**非該当理由をREADMEに記録**して受容（破壊的更新を避ける）。

**注意**: `frame-ancestors`/`sandbox`/`report-uri` は `<meta>` では無効（HTTPヘッダ専用）。
**人手**: 不要（配信ヘッダ設定のみ運用側）。

## 観点5: 色覚 / アクセシビリティ

**チェック**
- **色だけで情報を区別していないか**（赤緑、近接色）。線種・形・ラベル・記号で冗長化。
- canvas（Chart.js等）は**SRに不可視** → `role="img"`＋要約 `aria-label`、
  または正データを**テーブルで併設**。
- キーボードのみで全操作可能か、`:focus-visible` が見えるか。
- 小さい文字のコントラスト（WCAG AA）。チャート内の**ハードコード色**も対象。
- セマンティクス：`role="tab/tablist"`、`aria-selected`、フォーム要素の `aria-label`。

**対策（本PJ実例）**
- グラフ線を色＋**線種**で区別、凡例 `usePointStyle`、数学的な縦順も手がかりに。
- canvasに要約 aria-label、資産推移は年次テーブルが代替。
- 赤字警告に **▼マーク＋title** を併用（色非依存）。
- `:focus-visible` リング、`.sr-only`、`prefers-reduced-motion`。

**人手**: **実機 VoiceOver/TalkBack・色覚シミュレータ**での目視は必須（AIは断定不可）。

## 観点6: パフォーマンス

**手順：測ってから直す。**
- 重い同期計算（モンテカルロ等）の**1回あたり時間**を計測（16ms/フレーム基準）。
- スライダー等の**高頻度更新で毎回再計算**していないか（メインスレッド占有→ジャンク）。

**対策（本PJ実例）**
- 計測（約66ms/再計算）→ **入力デバウンス**（手を止めてから計算、操作値は即時更新）。
- 「更新中…」を `aria-live` で通知。初回は即時計算（デバウンス初期値）。
- さらに必要なら **Web Worker** 化（同一オリジンWorkerは `'self'` CSPで可）。
- バンドル：重いライブラリ（xlsx等）は**動的import**で分割。

**人手**: 不要。

## 観点7: UX / 情報設計 / コピー

**チェック**
- 初見で「何をして何が分かるか」想起できるか。結論（ヒーロー）→詳細の階層。
- 入力のしやすさ：単位の明示、スライダー＋ステッパー＋直接入力、即時フィードバック。
  数値入力が**タイピング途中状態（"2." 空欄）と喧嘩しない**か（ドラフト保持）。
- **表記揺れ**（タブ名と本文の不一致、全角/半角括弧、用語の揺れ）。
  ```
  grep -rn -E "\(|（" src --include=*.tsx   # 括弧の混在チェック
  ```
- 難解語（専門用語・ギリシャ文字・隠語）の平易化。
- 空欄表示バグ：`NaN`/`undefined` を数値入力に渡すと空欄化 → 安全値に矯正。

**人手**: 実ユーザーの**理解度テスト**は代替不可（特に確率・分位の解釈）。

## 観点8: テスト戦略

- **純関数（lib）**：仕様値の検算＋**不変条件（プロパティベース）**。
- **UI/結合**：`@testing-library/react`＋jsdom で主要操作（タブ切替・トグル・
  入力→結果反映・永続化往復・エクスポート呼び出し）。canvas/重い生成は**モック**。
  - Vitestは `globals:true` でRTLの**afterEach自動クリーンアップ**が効く（無いと
    renderが積み重なり「found multiple」になる）。
  - 既定 `environment:'node'`＋UIファイル冒頭に `// @vitest-environment jsdom`。
- 回帰の固定：**バグを見つけたら、その入力をテスト化**してから直す。

---

## 観点9: 運用・堅牢性・仕上げ

- **CI**（GitHub Actions）：push/PRで `lint → tsc → test → build` を自動実行。
- **ErrorBoundary**：描画例外で真っ白を防ぎ、再読込/データ消去の復旧導線。
- エクスポート等の失敗時の**ユーザー向けエラー表示**。
- **ドキュメント**：計算方法（methodology）、README、**LICENSE**（公開リポジトリ）。
- 仕上げ：ハードコード色のCSS変数化、`lang`/メタ、未使用コード削除。

```bash
npx tsc --noEmit && npm test && npm run lint && npm run build
```
